# RoutePilot — Agent Notes

RoutePilot is a driver onboarding & financial profile platform for gig/delivery drivers. This file is read automatically by the coding agent before each ticket, and the agent appends to the log below as it learns things — keep it accurate and short; prune stale entries rather than letting this grow forever.

## Stack & conventions

- Plain Node.js, ESM (`"type": "module"` in package.json). No web framework is wired up yet — check `package.json` dependencies before assuming one exists, since a "Basic Platform Setup" ticket may have added one since this was written.
- Tests: Node's built-in test runner (`node --test`), not Jest/Mocha/Vitest. Run with `npm test`. Convention is one `test/<name>.test.js` file per `src/<name>.js` module.
- `src/index.js` is the public entry point — it re-exports the pieces other code is meant to consume. When a ticket adds a new module other code should use, add its exports here too.
- Error handling convention: each module defines its own `<Name>Error` class (e.g. `AuthError`, `WizardError`, `JwtError`, `SessionError`, `BiometricError`) rather than throwing a plain `Error` — follow this pattern for new modules.
- Existing modules (as of when this was written): `auth.js`, `biometrics.js`, `dsp.js`, `encoding.js`, `jwt.js`, `onboarding.js`, `password.js`, `route-sync.js`, `session.js`, `tax-residency.js`, `totp.js`, `vehicle-lookup.js`, `vehicles.js`.

- HTTP server (`src/server.js`, `createServer()`) is built on Node's built-in `http` module, not Express — the repo had zero dependencies before this, so a hand-rolled router keeps that true. Revisit if routing needs grow (path params, middleware).
- `createServer()` only builds the server; it self-starts (reading `PORT`, default 3000) only when run directly via `node src/server.js` / `npm start`, guarded by an `import.meta.url` check. Import `createServer` from `src/index.js` for tests/composition instead of spawning a real process.
- Route handlers respond with `sendJson(res, status, body)` — keep new routes returning JSON via that helper for consistent headers.
- `createServer(config)` accepts optional `sessionManager`/`authService`/`now` overrides (defaults build an in-memory `createSessionManager` + `createAuthService`, secrets from `SESSION_ACCESS_SECRET`/`SESSION_REFRESH_SECRET`/`AUTH_CHALLENGE_SECRET` env vars with dev fallbacks). Tests inject their own pair for deterministic clocks — see `test/server.test.js`.
- Auth endpoints: `POST /register` → 201 `{ user }`; `POST /login` → 200 `{ status: 'authenticated', user, tokens }` or `{ status: 'mfa_required', mfaToken }`; `POST /login/verify-totp` → 200 `{ status: 'authenticated', user, tokens }`. `AuthError` codes are mapped to HTTP status via the `AUTH_ERROR_STATUS` table in `server.js` (400 default, 401 for bad credentials/MFA, 409 for duplicate username).
- `GET /dashboard` requires `Authorization: Bearer <accessToken>`, verified via `sessionManager.verifyAccess` — missing/invalid/expired all return a uniform 401. Returns `{ profile: { id, username, mfaEnabled, biometricEnrolled, createdAt } }`, built in `server.js` rather than reusing `auth.js`'s internal `sanitize()` (not exported) — keep both in sync if the user shape changes.
- `shifts.js` (`createShiftTracker()`) follows the same registry pattern as `dsp.js`: per-driver `Map` of records, frozen snapshots via `structuredClone` + `deepFreeze`, injectable `now`/`generateId`/`store`. Only one shift may be `active` per driver at a time (`SHIFT_ALREADY_ACTIVE`). Location is always client-supplied `{ lat, long }` — the server never geolocates — validated by a local `validateLocation` helper. Later shift-logging tickets (breaks, wait time, GPS trip) should extend this module's record shape rather than create a parallel per-driver store.
- Breaks/wait periods live on the active shift's `breaks`/`waits` arrays (`startBreak`/`endBreak`/`startWait`/`endWait` in `shifts.js`), each period `{ id, startedAt, endedAt, durationMs }`. Only one open period per type per driver at a time. `getDurations(driverId, shiftId)` sums `durationMs` across a shift's periods — still-open periods count as 0 until ended.
- Trip/mileage data also lives on the shift record (`shift.trip`), not a separate module: `addGpsPoint` accumulates `gpsDistanceMiles` via haversine distance between consecutive points; `setOdometer(driverId, { startMiles, endMiles })` records a manual reading. `getTripDistance(driverId, shiftId)` prefers the odometer reading over GPS when both are present — the driver's manual entry is treated as ground truth.
- `fuel.js` logs fuel purchases and EV charging sessions (`createFuelLogger()`, same per-driver `Map`-of-records pattern as `shifts.js`/`dsp.js`). Every record is normalized at write time: currency amounts to `BASE_CURRENCY` ('USD') via `convertCurrency`/`DEFAULT_EXCHANGE_RATES` (static illustrative rates, not live — pass `rates` to override), fuel volume to liters via `convertVolume`. Both converters are also exported standalone for reuse (e.g. future cost-per-mile / anomaly-check tickets) without needing a logger instance.
- **OCR provider interface** (`receipts.js`): `{ recognize(input): Promise<{ text?: string, fields?: object }> }`, where `input` is `{ buffer, path, mimeType }`. A real provider (Textract, Google Vision, Tesseract, …) implements `recognize`; `createMockOcrProvider({ responses, defaultResponse })` is the stand-in for now, keyed by upload `path`. A future ticket wiring in a real OCR service should only need to write a new object satisfying this interface and pass it as `ocrProvider` to `createReceiptProcessor` — no queue/extraction logic should need to change.
- `createReceiptProcessor()` queues uploads (`queue()`) onto a global FIFO (`pending`) plus a per-driver `Map` of records; `process`/`processNext`/`processAll` run them through the OCR provider and call `extractFields` to fill `{ vendor, date, total, currency }` — provider-returned `fields` win, missing ones are parsed from `text` via regex heuristics. A provider throwing marks the receipt `failed` (with `error`) rather than rejecting the whole sweep.
- `expenses.js` (`createExpenseTracker()`) maps a driver's expense category to a tax bucket via `bucketFor(categoryId, authority)`, reusing `tax-residency.js`'s `TAX_JURISDICTIONS`/`authority` model (IRS/HMRC) rather than redefining jurisdictions — `jurisdiction` on `categorize()` accepts a raw jurisdiction id ('US'/'GB'/'UK' alias), an authority string, or a full `declareTaxResidency()` declaration. `EXPENSE_CATEGORIES` bucket labels are a simplified approximation of real Schedule C / SA103 line items, documented as such in the module header — not tax advice. `resolveAuthority()` is exported from `expenses.js` (not duplicated) — `tax-estimation.js` and future tax-adjacent modules should import it rather than re-deriving authority from a jurisdiction.
- `tax-estimation.js` (`createTaxEstimator().estimateTax({ grossIncome, jurisdiction, expenses })`) is income tax only — no US self-employment tax or UK National Insurance, no state tax, UK personal allowance not tapered for high earners; all documented as assumptions in the module header. 2024 US federal single-filer brackets + standard deduction, 2024/25 UK England/NI bands (personal allowance modeled as a 0% bracket, not a separate deduction). Deductions are just `sum(expenses[].amount)` from `expenses.js` categorized records — no currency conversion, caller must pass matching-currency expenses. `computeProgressiveTax(income, brackets)` is exported standalone as the generic marginal-bracket calculator.
- `estimated-payments.js` hardcodes due-date schedules per authority (IRS: the 4 standard US estimated-tax dates; HMRC: the 2 Self Assessment payment-on-account dates, Jan 31/Jul 31 — modeled with the same quarter/countdown/payment machinery even though it's not literally quarterly). No weekend/holiday roll-forward. `nextDueDate()` searches ±1 tax year around `now` so it stays correct right after a year boundary. `createEstimatedPaymentTracker()` allows multiple (partial) payments per tax-year/quarter (`totalPaid` sums them) — same per-driver `Map` pattern as the other trackers.
- **Embedding provider interface** (`embeddings.js`): `{ embed(input: object): Promise<number[]> }`. `createMockEmbeddingProvider()` is the stand-in — it packs numeric mileage/earning fields (`totalMiles`, `totalEarnings`, `shiftHours`, `deliveries`) directly into a vector rather than calling a real model. A future ticket wiring in a real embedding model should only need to write a new object satisfying this interface and pass it as `embeddingProvider` to `createDriverPatternIndex` — the vector store and search logic don't change.
- `createVectorStore()` is an in-memory (`Map`) cosine-similarity nearest-neighbor store — chosen over file-backed to match every other module's store convention here (inject a different `store` for persistence later, same pattern as the trackers). `createDriverPatternIndex()` composes an embedding provider + vector store, tagging each stored vector with `driverId` so `findSimilarPatterns` can scope search to one driver's own history via the store's generic `filter` option.
- **LLM provider interface** (`tax-assistant.js`): `{ complete(messages: {role,content}[]): Promise<string> }` — the standard chat-completion message shape. `createMockLlmProvider({ reply })` is the stand-in (`reply` can be a string or a `(messages) => string` function for dynamic mocked responses in tests). A future ticket wiring in a real LLM should only need a new object satisfying this interface, passed as `llmProvider` to `createTaxAssistant`.
- `assembleContext()` in `tax-assistant.js` is the context-assembly step, kept separate from the LLM call so it's synchronously testable: it reuses `expenses.js`'s categorized expense records (`categoryLabel`/`bucket`/`amount`/`currency`) and `resolveAuthority` rather than re-deriving jurisdiction/bucket logic. `createTaxAssistant({ expenseTracker })` will auto-fetch a driver's expenses via `expenseTracker.list(driverId)` if `answerQuestion` isn't given an explicit `expenses` array.
- `cost-anomaly.js`: a cost is flagged when it's more than `DEFAULT_ANOMALY_THRESHOLD` (2) population-standard-deviations above the driver's own historical mean for that route — documented in the module header as a standard, conservative outlier cut. Fewer than 2 historical points, or zero-variance history, never trip a false positive (handled explicitly, no NaN/Infinity leaking out except a deliberate `Infinity` zScore when stddev is 0 and the cost is higher). `createRouteCostTracker()` keys history by `driverId::routeKey` — same in-memory `Map` convention as the other trackers.
- **Push provider interface** (`notifications.js`): `{ send(notification): Promise<{id, status}> }`. `createMockPushProvider()` is the stand-in (records sent notifications on `.sent`). A future ticket wiring in real push (FCM/APNs/a notification service) only needs a new object satisfying this interface, passed as `pushProvider`. Trigger conditions are pure/exported standalone for direct testing: `shiftMissingMileage(shift)` (reuses `shifts.js`'s `shift.trip` — no odometer and no positive `gpsDistanceMiles`) and `receiptOverdue(purchaseAt, receipts, { windowMs, now })` (default window `DEFAULT_RECEIPT_WINDOW_MS` = 48h, reuses `receipts.js` records' `queuedAt`). `createNotificationScheduler().sweepDriver()` composes both against a `shiftTracker`/`receiptProcessor` and sends via `pushProvider`.
- `analytics.js` splits weekly gross/net profit into `bucketWeeklyProfit({earnings, expenses})` (pure data bucketing by ISO week, the only part that's meaningfully unit-tested) and `renderWeeklyProfitChartSvg(buckets)` (hand-rolled SVG, no charting dependency). There's no persisted earnings-log module in this repo yet, so both the function and the new `POST /dashboard/weekly-profit` route (session-protected, same `requireSession` helper `handleDashboard` now also uses) take `earnings`/`expenses` as `[{at, amount}]` directly from the caller rather than looking them up server-side.
- `payout-reconciliation.js` compares dsp.js's `computePayout(link.payoutRate, route.work)` ("recorded" earnings) against route-sync.js's synced `route.earnings` ("reported" by the DSP portal) — no new DSP/route data model, just the diff. **Important**: `reconcileRoute(route, payoutRate)`'s `payoutRate` must be a *validated* card (from `validatePayoutRate`/a stored link's `.payoutRate`), not a raw input — `computePayout` relies on the validated shape's `basis` field per component, and silently computes 0 against an unvalidated one (caught this via a failing test). Mismatch tolerance defaults to `DEFAULT_RECONCILIATION_TOLERANCE` (1 cent, absolute). Not yet wired to an HTTP route — module only, same as ticket scope.
